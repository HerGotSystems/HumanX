## Purpose

Audit F-5 (packet-ID storage) from the D-268A friction finding list before any implementation. Determine whether `packet_id` from a RunPack can be safely preserved when saving AI analysis results, and whether this can be done frontend-only or requires backend/API/storage/schema work.

---

## Audit Findings

### Q1. What does `saveAnalysisResult()` currently POST to `/api/analysis`?

**`public/app-v10.js` line 573:**
```javascript
await api('/api/analysis',{method:'POST',body:JSON.stringify({claimId:selected.id,source:'runpack-user',raw:result})})
```

Current POST body fields:
- `claimId` — selected claim ID
- `source` — hardcoded `'runpack-user'`
- `raw` — the full AI return text (string)

**`packet_id` is NOT included in the POST body.** Neither `source_claim_id`, `source_snapshot_hash`, nor `generated_at` are forwarded either.

---

### Q2. What does `addAnalysisResult()` in `src/analysis-results.js` currently accept and store?

The handler reads `body.claimId || body.claim_id`, `body.analysis || body.result || body.raw`, and `body.source`.

The INSERT is:
```sql
INSERT INTO analysis_results (
  id,claim_id,user_id,source,verdict,evidence_score,testability,survivability,
  strongest_support_json,strongest_pressure_json,missing_tests_json,
  plain_language_summary,raw_json,created_at
) VALUES (?,?,?,?,?,?,?,?,?,?,?,?,?,?)
```

**`packet_id` is NOT in the INSERT.** The function does not read `body.packet_id` at all.

---

### Q4. Where does `packet_id` currently exist in the frontend session state?

`packet_id` is generated by `buildProvenanceMeta(claim)` via `generatePacketId(claim.id)` (line 563–565) and stored as a field in the `lastPacket` JSON string. It is available from `JSON.parse(lastPacket).packet_id` after any packet generation.

For backend packets, `packet_id` is returned from the backend `createAipPacket()` worker (line 897) as `data.packet.packet_id`, merged into `lastPacket` during `generateRunPack()`.

---

### Q5. Is `packet_id` ever forwarded to the backend today?

**No.** The advisory packet-ID mismatch check in `saveAnalysisResult()` reads `parsed.packet_id` (from the AI return JSON) and `JSON.parse(lastPacket).packet_id`, compares them, and shows a toast on mismatch — but neither value is ever sent to the API.

---

### Q6. What value of `packet_id` should be stored with the analysis result?

The correct value to store is `JSON.parse(lastPacket).packet_id` — the `packet_id` of the packet that was given to the AI, not the `packet_id` parsed from the AI return. The AI return `packet_id` may be absent, stale, or from a different session.

The frontend already has this value available in `lastPacket` at the moment `saveAnalysisResult()` is called. It is also available in `lastPacketMeta.packet_id` (set in `generateRunPack()` as `lastPacketMeta = provenance`).

---

### Q7. Is this change frontend-only?

**No — this requires backend/schema/API changes.** The three required changes are:

1. **Schema migration** — new `packet_id TEXT` column on `analysis_results` table (new migration file, e.g. `0017_analysis_results_packet_id.sql`)
2. **Backend handler** — `addAnalysisResult()` in `src/analysis-results.js` must read `body.packet_id` and pass it to the INSERT
3. **Frontend** — `saveAnalysisResult()` in `public/app-v10.js` must include `packet_id` from `lastPacketMeta` in the POST body

All three changes are required together. Deploying the frontend change alone will silently drop the `packet_id` (the backend will ignore an unknown field). Deploying the backend change without the migration will fail (column does not exist). The migration must land first.

**Per workflow rules, this requires branch/PR style.** It is not a docs-only or frontend-only change.

---

### Q8. What is the smallest correct implementation path?

**Step 1 — Migration (new file):**
```sql
ALTER TABLE analysis_results ADD COLUMN packet_id TEXT;
```
One `ALTER TABLE` statement. No default needed — `NULL` is acceptable for historical rows.

**Step 2 — Backend handler (`src/analysis-results.js`):**
- Read `body.packet_id` — apply the same `cleanId()` helper used for `claimId` (or `cleanText(..., 80)` if the format allows non-ID characters)
- Add `packet_id` to the INSERT column list and bind list
- Add `packet_id: a.packet_id || null` to `mapAnalysis()` return

**Step 3 — Frontend (`public/app-v10.js`, `saveAnalysisResult()`, line 573):**
- Read `lastPacketMeta?.packet_id` (already available from `generateRunPack()`)
- Include it in the POST body: `{claimId:selected.id,source:'runpack-user',raw:result,packet_id:lastPacketMeta?.packet_id||null}`

No new helper functions needed. No new routes needed. No new UI needed. No CSS changes needed.

---

### Q9. What is the `packet_id` format and is `cleanId()` safe to use on it?

Frontend packet IDs are generated by `generatePacketId(claimId)`:
```javascript
function generatePacketId(claimId){const ts=Date.now().toString(36);const short=(claimId||'').slice(-6);return'rp_'+short+'_'+ts}
```
Example output: `rp_abc123_lc2x9k` — contains `rp_`, alphanumeric, and underscores.

Backend packet IDs use `makeId('rp')` — same prefix convention.

`cleanId()` in the worker strips non-alphanumeric chars (typically `/[^a-z0-9]/gi`). The underscores in `rp_*` IDs would be stripped, corrupting the value. Use `cleanText(body.packet_id || '', 80)` instead, or a dedicated regex that allows `rp_` format characters. This detail must be confirmed when reading `cleanId()` in `worker.js` before implementation.

---

### Q10. What happens to existing `analysis_results` rows after the migration?

Existing rows will have `packet_id = NULL`. That is correct — they were saved before packet-ID storage was implemented. `mapAnalysis()` should return `packet_id: a.packet_id || null` so callers receive `null` for old rows rather than `undefined`.

---

### Q11. Will the advisory packet_id mismatch toast still work after D-275B?

Yes — the advisory check compares `parsed.packet_id` (from AI return) against `JSON.parse(lastPacket).packet_id` (current session packet). This runs before `saveAnalysisResult()` stores the result. The advisory is unchanged by D-275B; it remains non-blocking.

---

### Q12. Should `source_claim_id`, `source_snapshot_hash`, or `generated_at` also be stored?

Not in D-275B. F-5 scope is `packet_id` only. The other provenance fields (`source_claim_id` equals `claimId` which is already stored, `source_snapshot_hash` and `generated_at`) can be added in a follow-up if needed. Expanding scope to all provenance fields would require additional migration columns and is a separate decision.

---

### Q13. Is there a race condition risk — what if `lastPacketMeta` is stale?

`lastPacketMeta` is set once at packet generation time inside `generateRunPack()`. It is not cleared between claims or on session events. If the user generates a packet for claim A, switches to claim B without generating a new packet, then saves an analysis on claim B, `lastPacketMeta.packet_id` would be from claim A's packet.

**Mitigation:** The frontend should only forward `packet_id` when `lastPacketClaimId === selected.id` — the same gate already used for auto-expand and "Recreate Packet" display. Implementation in D-275B:
```javascript
const pktId = (lastPacketMeta && lastPacketClaimId===selected?.id) ? (lastPacketMeta.packet_id||null) : null;
```

---

### Q14. Public profile impact

`saveAnalysisResult()` is only called from the authenticated Study view. The `/api/analysis` route already requires authentication (`requireUser`). Adding `packet_id` to the POST body and the stored row does not expose any new data on the public profile. `listAnalysisForClaim()` is not called from any public route. `mapAnalysis()` adding `packet_id` to its return will need to be checked against any public API handler that calls it — if none call it publicly today, no change needed.

---

### Q15. Review/moderation impact

None. `addAnalysisResult()` is not called from any Review/moderation handler. `analysis_results` rows are not read by `reviewDecisionUI`, `requestApproveReview`, or `requestRejectReview`.

---

### Q16. Drift/Belief expansion impact

None. `public/belief-drift-expansion.js` does not reference `saveAnalysisResult()`, `/api/analysis`, or `analysis_results`.

## Feasibility Verdict

**F-5 requires backend/schema work.** This is NOT frontend-only. Three changes must be coordinated: migration, handler, frontend. Must be done via branch/PR style per workflow rules.

**Estimated scope:** Small. One `ALTER TABLE` migration, ~5 lines in `analysis-results.js`, ~2 lines in `app-v10.js`. No new routes, no new helpers, no UI changes.

**Risk:** Low. `packet_id` is additive. Existing rows get `NULL`. Advisory toast logic is unchanged. No moderation or public profile surface is affected.

**Recommended next task: D-275B** — implement F-5 via branch. Prerequisites: confirm `cleanId()` vs `cleanText()` for packet-ID sanitization (Q9 above); add migration; update handler; update frontend; 20+ regression tests.
